External Hackathon Challenge with ASDK

Introduction

The Ather 450X consists of many ECUs for different roles like battery management, motor control, body control and dashboard display.

Body Control Module communicates with BMS, Charger, Motor-Controller and Dashboard ECUs to maintain vehicle-states.

Example :

  • Key On to Riding

  • Charging

Body Control Module controls lamps, switches and other peripherals in the vehicle.

Example :

  • Read Indicator switch and Blink Left and Right Indicators lights

  • Brake Lamp light On/Off based on Brake switch

  • Horn On/Off with Horn switch

  • Low beam and High beam lamp drive

Problem Statement

Write an firmware application for the Body Control Module of Ather 450X, using ASDK stack for Traveo T2G Starterkit based on CYT2B75 microcontroller.

Hackathon Tasks

NibbleNinja

  1. Demo Task:

    1. Read High Beam Switch

    2. Once Switch is Turned On (using interrupt method) then enable HighBeam

    3. Once Switch is Turned Off then disable HighBeam

  1. Control Horn

    1. Read Horn Switch

    2. Once Switch is Pressed (using Interrupt method) then enable Horn output

    3. Once Switch is Released (without using interrupt (polling)) then disable Horn output

  1. Control Indicators

    1. Read Indicator-Left Switch, Right Switch and Centre Switch

    2. Once pressed turn on respective LED, and turn off other LED
      Example: If Left_Indicator switch is pressed then turn_On left_indicator_LED and turn_Off right_indicator_LED

    3. Once released, dont change LED

    4. If Centre Switch pressed, then turn off both LEDs

  2. Read Brakes and control TailStop Lamp intensity

    1. Read Brake1 & Brake2 Switches

    2. Set TailLamp as PWM (period 1kHz)

    3. Keep TailLamp DutyCyle 20% when no brake is pressed

    4. For either brakes presssed, set TailLamp DutyCycle to 100%

ByteBrain

CAN communication via on-board CAN transreciver of Traveo StarterKit connected to CAN module 0, channel 1. Initialize CAN with following configuration and run below tasks.

  1. Publish all Peripheral Inputs into CAN 0x401 as per protocol

  2. Publish all Peripheral Outputs into CAN 0x402 as per protocol using Elektrobit stack. Refer Elektrobit section.

  3. Parse vehicle information from CAN 0x501 as per protocol using Elektrobit stack. Refer Elektrobit section.

KiloCoder

  1. Turn On Motor

    1. Peripheral Requirements to be checked from CAN message 0x501

      1. Sidestand Disengaged

      2. Charger not connected

      3. Key turned On

      4. Kill Switch (Unkilled state)

      5. BMS SoC more than 10%

      6. Motor Temperature less than 50 degrees

    2. Look for BRAKE & START switch pressed together

    3. Publish CAN 0x403 with MotorOn signal once all condition is met using Elektrobit stack. Refer Elektrobit section.

    4. Turn Off motor if one of the “Peripheral Requirement” fails

  2. Motor Mode Change

    1. Read existing Motor Mode from 0x501

    2. Read Left/Right Mode switch

    3. Motor Mode on pressing Right Mode Button : Eco → Ride → Sport → Warp

    4. Motor Mode on pressing Left Mode Button : Warp → Sport → Ride → Eco

    5. Verify Mode from motor on CAN 0x501

    6. Request next mode on 0x403

  1. Play Sound from the motor using user inputs from switches

    1. Keep Motor-On

    2. Assign some switch to music Keys C, C#, D, D#, E, F, F#, G, G#, A, A# and B (except HORN and KILL switch)
      Example : BRAKE1 switch -→ assigned to C
      INDICATOR_LEFT switch -→ assigned to C#

    3. On button press, request the sound from motor_controller as per protocol on CAN 0x606

  2. Play the following Song and Light on 450X

    2. Play Each note of the below sequence for 200ms by sending 606 can message with appropriate data as per Table:5

    3. ”–” is equal to 200ms delay (i.e not to send 0x606 can message)

      Music Sequence :

      E – D# – E – D# – E – B – D – C – A –––

      F – G – A – B –– F – G# – B – C –––

      E – D# – E – D# – E – B – D – C – A –––

GigaGuru

  1. Read LDR ADC input and control eval board User LED 4 intensity

    1. Convert 12bit raw ADC into 0-100

    2. Set User LED4 as PWM (period 1.5kHz)

    3. Control User LED 4 intensity using PWM with above value as dutycycle

  2. Blink Indicators

    1. read indicator switch and blink respective indicator

    2. Blink pattern : LED On for 300ms, LED Off for 300ms

  3. Enter and Exit Park Assist mode

    1. Keep vehicle in motor-On condition

    2. Press both Brakes and Center Indicator for 3 seconds

    3. Read the above condition and request for PARK_ASSIST entry/exit

    4. Requesting mode on 0x403 as per protocol

      Ex: If vehicle is in SPORT mode, and BRK+INDICATOR_CENTER is detected then request mode change to PARK_REVERSE
      if vehicle is in PARK_REVERSE mode, and BRK+INDICATOR_CENTER is detected then request last saved mode (SPORT mode in this case)

  4. Implement Hazard Lights

    1. Verify the vehicle inclination w.r.t. Roll/Pitch/Yaw from CAN 0x502

    2. If one of the angle is +/-60 then trigger Hazard lights

Transmitting CAN Messages

CAN messages 0x402 and 0x403 are sent out automatically. To update the respective bytes in these CAN messages use the following snippet:

Com_SendSignal(<SIGNAL_NAME>, &<SIGNAL_DATA>);

Refer the below tables for SIGNAL_NAMEs for 0x402 and 0x403:

uint8_t mode = 3;
Com_SendSignal(MODE_CHANGE_REQUEST, &mode);

Receiving CAN Messages

CAN messages 0x501 are parsed automatically. But to receive them, please add below code snippet in your CAN Callback function.

if (ASDK_CAN_RX_EVENT == event)
{
        switch (can_message->can_id)
        {
        case 0x501:
            Can_HwType mailbox;
            PduInfoType PduInfo;

            mailbox.CanId = can_message->can_id;
            mailbox.ControllerId = 0;
            mailbox.Hoh = 0;

            PduInfo.SduLength = can_message->dlc;
            PduInfo.SduDataPtr = can_message->message;
            CanIf_RxIndication(&mailbox, &PduInfo);
            break;

        case 0x502:
            /* user code */
            break;

        default:
            break;
        }
}

In above code, write your own logic for parsing 0x502.

Once 0x501 message is received and parsed, the bytes information from the message is updated in their respective variables. These variables can be accessed by including the header file “system.h”.
